## Orchestration Contract

This skill is a PURE ORCHESTRATOR. It coordinates agents — it does NOT touch CSS files, write to the database, or run compliance checks itself. Every piece of real work is delegated to a specialist agent. Clean boundaries. Tremendous architecture.

- **Token extraction and writes** -> `token-manager` agent
- **Token addition and validation** -> `token-manager` agent
- **Component catalog and scope exploration** -> `scout` agent (read-only)
- **Drift detection and compliance scanning** -> `token-auditor` agent

The skill's job: receive the user's intent, select the right mode, spawn the right agents with the right inputs, and present results. That's it. No side work. No freelancing. DISCIPLINE.

## Modes

### init

Initialize a design system from existing project styles.

1. Identify CSS/style source file paths in the project
2. **Spawn `token-manager` agent** with the source file list -> the agent extracts tokens, validates them against security allowlists, writes to DB via `mpga` CLI, and generates `tokens.css` + `tokens.json`
3. Present the agent's summary: token count by category, any rejected values, generated file paths

### add-token

Add a single design token to the registry.

1. Collect the token spec from the user: `name`, `value`, `category`
2. **Spawn `token-manager` agent** with the token spec -> the agent validates the name (kebab-case), validates the value against security allowlists, writes to DB, and regenerates output files
3. Present confirmation or rejection reason from the agent

### catalog

Inspect the component catalog and detect token drift.

1. **Spawn `scout` agent** in read-only mode targeting the design system scope -> returns the component list with evidence links
2. **Spawn `token-auditor` agent** -> returns the drift report:
   - **Used-but-undocumented**: raw values appearing in 3+ files with no corresponding token
   - **Documented-but-unused**: tokens in the registry with zero usages in scanned files
3. Present the catalog with drift flags from both agents

### audit

Scan source files for hardcoded values and report token compliance.

1. Collect scope list or file list from the user
2. **Spawn `token-auditor` agent** with the scope/file list -> the agent scans for hardcoded hex colors, px/rem values, inline fonts, magic breakpoints, and reports compliance percentage with per-file scores and suggested token replacements
3. Present the compliance report with:
   - Per-file compliance scores and violation details
   - Overall compliance grade (TREMENDOUS / GOOD / NEEDS WORK / SAD)
   - Suggested fixes with specific token variable names

## Token Format

Tokens are organized into four categories:
- `color` — hex, rgb, hsl values
- `spacing` — px, rem, em values
- `typography` — font families, sizes, weights, line heights
- `breakpoint` — numeric pixel values for media queries

## Security

The `token-manager` agent handles ALL value validation before any database write. It enforces:
- **Allowlist**: hex colors, functional colors (`rgb()`, `hsl()`), px/rem/em values, quoted font names, numeric breakpoints
- **Blocklist**: `url()`, `expression()`, `@import`, `javascript:`, `data:` URIs, values containing `;`

No token reaches the database without passing validation. The skill never validates directly — it trusts the agent's strict protocol. LAW AND ORDER.

## Output

- Updated design tokens in the DB (`.mpga/mpga.db`) — written by `token-manager`
- Generated `tokens.css` and `tokens.json` in `.mpga/` — generated by `token-manager`
- Component catalog with evidence links — produced by `scout`
- Compliance report with per-file scores and drift flags — produced by `token-auditor`
